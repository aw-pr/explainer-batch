SGLang speeds up LLM programs; it stops recomputing the same prefixes

This is not a new model. It is a Python DSL plus a cache-aware runtime that remembers shared prompt work, then reuses it across branches, agents, and long documents. The paper's practical claim is simple enough to trust: when an LLM app keeps seeing the same prefix, you should not make the GPU rediscover it from scratch.

What the system is

SGLang gives you a small programming language embedded in Python, with primitives such as gen, select, fork, and join. That sounds modest, which is often a sign that the authors have done the hard work somewhere else. In this case, the hard work sits in the runtime, where the system tracks prompt state and generation calls as a structured program rather than a pile of ad hoc API calls.

The paper's bet is that LLM applications have already moved past single prompts. They now branch, loop, call tools, and stitch together partial outputs. Once you admit that reality, a plain request-response API starts to look rather underdressed, so SGLang puts the structure into the language itself and lets the backend do the awkward bookkeeping.

Why it runs faster

The main trick is RadixAttention, which keeps a radix tree of the KV cache, the model's saved attention work for a prefix. If two requests share text at the start, SGLang reuses that work instead of running the same tokens through the model again. That matters most in few-shot prompts, multi-turn agents, and any workflow that keeps dragging the same context through a fresh call.

The runtime also batches requests, detects independent branches, and schedules work to keep cache hits high. The paper reports up to 6.4x higher throughput across a spread of tasks, 1.3x on generative agents, 1.7x on LLaVA-Bench, and 2.9x on a long-context JSON task. The pattern is boring in the best way: the more repeated structure the app has, the more waste SGLang can cut away.

Why it matters

SGLang sits between prompt libraries and inference engines, which means it can express the program and optimise the execution in one place. That is useful because the two layers usually ignore each other, then complain about latency in separate meetings. The paper shows that the combined approach buys both control and speed, instead of asking you to choose one and pretend the other will be fine.

The case studies make the point nicely. The authors add a vision-language primitive in around 400 lines of code, run it on GPT-4 Vision without code changes, and show that GPT-4 can reorder 12 of 15 prompt templates safely while lengthening the shareable prefix by 60 tokens on average. In other words, SGLang does not just accelerate the common case. It also makes the uncommon case less awkward, which is usually where useful systems earn their keep.
