Research Explainer · Hong et al. (2025)

Splitting GPU compute without splitting GPU memory cuts LLM latency, while using half the hardware

semi-PD keeps the prefill and decode phases of LLM inference on separate compute lanes within the same GPU, eliminating the storage waste and transfer overhead that plague fully disaggregated serving systems.

The Problem

Every LLM request has two phases. The prefill phase digests the entire prompt and produces the first token. The decode phase then generates tokens one at a time. These phases have wildly different computational profiles: prefill is compute-heavy, decode is memory-bandwidth-heavy. When you run both on the same GPU (a "unified" system), they fight over resources, inflating latency for one phase whenever the other is busy.

The industry's answer has been to disaggregate the two phases onto separate GPUs. Systems like DistServe and Splitwise dedicate some GPUs to prefill and others to decode. This eliminates interference, and it works, until you look at what it costs in storage. The decode GPU stores all the KV cache (the running memory of the conversation). The prefill GPU stores almost none, because it ships its KV cache over to the decode GPU the moment prefill finishes. The result: up to 89% of the prefill GPU's memory sits empty, while the decode GPU runs out and triggers latency explosions.

On top of that, you pay for a full copy of the model weights on every GPU. Deploying Llama3-8B in a disaggregated setup requires at least two GPUs. Deploying Llama3.1-405B becomes impossible on hardware that a unified system could handle. The paper's Figure 4 shows the consequences directly: once the decode instance's KV cache storage hits capacity, P90 TPOT jumps by roughly 3×.

The Insight

Hong and colleagues at Tsinghua/Infinigence-AI noticed something that seems obvious in hindsight. The real benefit of disaggregation is disaggregated computation (running prefill and decode asynchronously so they don't interfere). The real cost is disaggregated storage (duplicated weights, KV cache transfers, memory imbalance). So the fix is simple in principle: disaggregate the compute, unify the storage.

semi-PD does exactly this. Both the prefill worker and the decode worker live on the same GPU, sharing one copy of the model weights and one pool of KV cache memory. They run as separate processes with their own streaming multiprocessors (SMs) partitioned via NVIDIA's Multi-Process Service (MPS). Each worker gets a tunable percentage of the GPU's compute cores, specified as a pair (x, y), and they execute asynchronously without stepping on each other's latency.

How It Works

The system has two main components. A Computational Resource Controller partitions the GPU's SMs between the prefill and decode workers. An (x, y) = (60, 40) config means 60% of SMs go to prefill, 40% to decode. When workload patterns shift, the system adjusts the ratio dynamically, using an SLO-aware feedback algorithm that monitors real-time TTFT and TPOT against user-defined targets.

The trickier engineering is in the Unified Memory Manager. Both workers need to allocate KV cache blocks from a shared pool, and they do so asynchronously. This creates write-after-read conflicts: one worker queries the current memory utilization, then allocates blocks, but the other worker may have already changed the utilization between the query and the update. semi-PD wraps all three steps (query, get blocks, update) into an atomic operation, preventing memory accounting errors without adding meaningful overhead.

Resource adjustment is where semi-PD truly separates itself from disaggregated systems. DistServe's periodic replanning takes minutes (it has to reload model weights). TetriInfer's instance flip has to drain the request queue first. semi-PD introduces a resident process that permanently holds the weights and KV cache in GPU memory, sharing pointers to new worker processes via inter-process communication. Switching from (60, 40) to (50, 50) costs only the time to initialise the new MPS processes, not the time to reload a 70-billion-parameter model.

What They Found

The team benchmarked across Llama3-8B, Llama3-70B, Llama3.1-405B, DeepSeek-V2-Lite, and DeepSeek-V3, on A100, A800, and H200 GPUs, using ShareGPT, LongBench, and MATH-500 datasets. The headline numbers are stark. On DeepSeek models, semi-PD cuts average end-to-end latency by 1.27–2.58× compared to SGLang. On Llama models, it sustains SLO attainment above 90% at request rates where every baseline has already collapsed: 1.55× more requests than vLLM-S, 1.72× more than vLLM-D, 1.62× more than DistServe.

The P90/P99 latency charts tell the more interesting story. As request rates climb, vLLM variants see TTFT explosion (because decode requests starve prefill of resources) and DistServe sees TPOT explosion (because the decode instance runs out of KV cache memory). semi-PD's curves stay flat. On Llama3-8B with ShareGPT, semi-PD handles over 16 requests per second per GPU while every competitor has already hit a latency cliff.

The 405B model is particularly telling. DistServe simply cannot be deployed, because its weight replication demands more GPUs than the test cluster provides. semi-PD deploys it on 32 A100-40GB GPUs with TP=8 and PP=4, with room to spare. At the cluster scale, replacing some disaggregated instances with semi-PD instances (1P3D4S instead of 2P6D) eliminated TTFT surges under heterogeneous workloads and halved TPOT.

Why It Matters

LLM serving is becoming the dominant cost in AI infrastructure. Every percentage point of GPU memory wasted or every millisecond of unnecessary latency compounds across millions of requests per day. Disaggregated systems addressed a real problem (prefill-decode interference) but introduced four new ones: storage imbalance, KV cache transfer overhead, coarse-grained resource adjustment, and doubled weight storage. semi-PD eliminates all four while preserving the core benefit.

The practical implication is that smaller deployments, private or local setups with limited GPUs, can now run disaggregated-quality inference without needing twice the hardware. For cluster operators, semi-PD instances function as elastic buffers that absorb workload spikes, precisely because they can shift compute between phases in milliseconds rather than minutes.

The latency modelling is worth noting for anyone building serving schedulers. The paper shows that TTFT follows an M/M/1 queuing model (inversely proportional to SM ratio minus arrival rate), while TPOT is simply inversely proportional to its SM share. Both relationships fit with R² > 0.995. That is a clean, tractable model for dynamic resource allocation, and it works because the two workers are genuinely independent once you've unified the storage layer.
