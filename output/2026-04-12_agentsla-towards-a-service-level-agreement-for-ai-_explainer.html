Research Explainer · Jouneaux and Cabot (2025) · Preprint

Agent quality is not a model problem, it is a contract problem

AgentSLA argues that AI agents need service-level agreements that can be checked, enforced, and integrated with existing agent protocols. The paper extends ISO/IEC 25010 with AI-agent-specific quality characteristics, then turns that model into a JSON-based DSL and a validating Python parser.

ISO/IEC 25010+ A quality model extended with characteristics tailored to AI agents rather than generic software alone.

AgentSLA DSL A domain-specific language for specifying service-level agreements for AI agents in a machine-readable form.

Python parser A validating implementation that checks the JSON description before instantiating the agreement model.

Protocol fit Designed to work alongside agent communication protocols such as A2A, not replace them.

The Problem

The paper starts from a simple mismatch. AI agents are becoming part of real software systems, but the quality expectations around them are still underspecified. Traditional service-level agreement languages can describe service promises, yet they were built before agentic systems became common and do not capture the quality concerns that matter for agent behavior.

That creates a practical gap. A consumer of an AI agent may care about performance, availability, fault tolerance, response quality, and other operational properties, but those expectations are often implicit. AgentSLA frames this as a quality-assurance problem: if the contract is not explicit, it cannot be checked or enforced consistently.

Implementation Notes

The authors report a Python implementation that parses the JSON description, validates that the expected units and metric types are present, and ensures values fall within required bounds before instantiating the agreement model. They also describe modeling the metamodel in BESSER Low-Code so Python classes can be generated from the abstract syntax.

That makes the work more than a taxonomy exercise. It ties the quality model to executable tooling, which is the main reason the proposal matters for deployment and monitoring workflows.

Limits And Next Steps

The discussion section is careful: the quality model is still only partially validated in real agentic systems. The paper also notes that broader validation and more tool support are still needed before the language can be treated as mature infrastructure.

Even so, the direction is clear. If AI agents are going to be treated as production services, their quality assumptions need to be made explicit in a contract that software can inspect, validate, and enforce.

Key Implication

AgentSLA treats agent quality as a first-class engineering object. The contract is the product: define the agent's quality model, express it in a machine-readable DSL, validate it before use, and make it compatible with the protocols the rest of the system already speaks.
