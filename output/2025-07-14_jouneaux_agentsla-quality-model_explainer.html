Research Explainer · Jouneaux (2025)

AI agents need service contracts too, but nobody has written the language for them yet

AgentSLA proposes the first domain-specific language for specifying Service Level Agreements for AI agents, backed by a quality model that extends the ISO/IEC 25010 standard with agent-specific dimensions like sustainability, autonomy, and output properties.

Published November 2025

KEY CONTRIBUTION

This paper introduces AgentSLA, a JSON-based domain-specific language for writing formal Service Level Agreements for AI agents. It pairs this language with a new quality model that extends the ISO/IEC 25010 software quality standard, adding dimensions specific to AI agents (sustainability, autonomy, output properties, and understandability) along with 30+ concrete metrics mapped to those dimensions. A Python validating parser is provided as open-source tooling.

From Model-as-a-Service to Agent-as-a-Service

When software teams started plugging LLMs into their products, the dominant pattern was Model-as-a-Service: fire a prompt at an API, get a response, carry on. That model is giving way to something more ambitious. AI agents can plan, use tools, call other agents, and refine their own answers autonomously. The industry is moving toward Agent-as-a-Service, with protocols like Google's Agent2Agent (A2A) emerging to wire these agents into traditional software architectures.

The problem is that service-oriented architectures have always depended on formal quality contracts. Web services have had SLA specification languages (WSLA, SLAng, WS-Agreement) for two decades. Cloud computing, IoT, and security each have their own tailored formalisms. AI agents have nothing. No agreed quality model, no standard for expressing what "good enough" means, and no machine-readable way to say "this agent must respond within 500 milliseconds and explain its reasoning." Jouneaux and Cabot set out to fill that gap.

A quality model built on top of ISO 25010

The authors reviewed six prior quality models for AI software mapped to ISO/IEC 25010 and found them incomplete. Only two discussed sustainability. Two skipped explainability entirely. None addressed the interaction dynamics created by an agent's autonomy level. Rather than start from scratch, the paper extends the existing ISO standard, adding new sections and broadening existing ones.

Four new quality dimensions stand out. Sustainability covers training impact, inference impact, and mitigation strategies (carbon offsets, for example), drawing on the authors' own Sustainability Model Card work. Autonomy captures the degree of human oversight an agent requires, from fully supervised to fully independent. Output properties address the reality that two correct answers can differ wildly in conciseness, consistency, creativity, and diversity. And Understandability groups interpretability, transparency, explainability, and accountability into a single umbrella, extending the existing ISO accountability characteristic to cover traceability through AI decision processes.

Alongside the quality model, the paper provides a table of over 30 concrete metrics mapped to these characteristics. These range from well-known measures like precision, recall, and F1 score through to TTFT (Time-To-First-Token), SHAP estimation error, carbon emissions, and an OversightLevel metric based on recent work by Cihon et al. on measuring agent autonomy. Some characteristics, like creativity and diversity, are acknowledged to lack precise metrics, which is at least honest.

The AgentSLA language itself

AgentSLA is a JSON-based domain-specific language for expressing SLAs as machine-readable contracts. The abstract syntax (the metamodel) structures an SLA as a collection of guarantee terms. Each guarantee term has scopes (which agents it applies to), qualifying conditions (preconditions that must hold before the SLO kicks in), and one or more Service Level Objectives. An SLO is a boolean expression built from comparisons: a QoS metric on one side, an operator in the middle, a threshold on the other.

The clever part is how the language handles the nondeterminism inherent to AI agents. Each QoS metric carries an explicit uncertainty value, acknowledging that the same agent given the same prompt will not always produce the same latency or quality score. A DerivedQoSMetric allows aggregation over a time window (average TTFT over the last ten messages, for instance). And a QoSDriftMetric computes the difference between two consecutive windows, letting you detect when an agent's quality is degrading as its context evolves. This drift-awareness is genuinely novel for SLA languages.

The concrete syntax uses JSON so that SLAs can be exchanged directly within emerging agent communication protocols like A2A. A Python implementation, built using the BESSER low-code platform, provides a validating parser that checks unit correspondence, metric types, operator validity, and confidence bounds before instantiating the model. The authors envision using these parsed models to auto-generate monitors, select agents based on their advertised quality, or build orchestrators that swap agents dynamically when SLOs are violated.

What remains to be proven

The authors are upfront about the limitations. The quality model has been discussed with partners of the MOSAICO project but not formally validated in production agentic systems. The core metric set is fixed; users cannot currently define custom metrics, which limits applicability to domain-specific use cases. Some quality characteristics (creativity, diversity) have no known metrics at all. And while the JSON syntax was designed to fit alongside A2A, agents receiving an AgentSLA document today would simply ignore it because SLA handling is not part of the protocol standard.

The roadmap is ambitious: an extensible DSL that lets users define their own metrics, a graphical notation for less technical stakeholders, a conversational interface for defining SLAs through dialogue, and integration with Google's Agent Development Kit. The most compelling near-term application is automatic agent selection, where an orchestrator picks among equivalent agents deployed in different regions based on which one's advertised quality metrics best satisfy a given SLA. That is a real problem, and AgentSLA gives it a real specification language for the first time.
