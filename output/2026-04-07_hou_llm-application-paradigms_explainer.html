Research Explainer · Hou, Zhao & Wang (2025)

LLM apps are stuck in four incompatible silos, but a three-layer architecture could break them out

A vision paper maps the landscape of LLM application paradigms, from app stores and agents to self-hosted services and on-device AI, and argues that infrastructure, protocol, and application layers need to work together before the ecosystem can grow up.

The Landscape Today

The authors survey 20 LLM app stores (including OpenAI's GPT Store, ByteDance's Coze, Tencent's Yuanqi), 11 major agent frameworks (LangChain alone has 116k+ GitHub stars), and a fast-growing self-hosted ecosystem where over 400,000 Ollama instances were detected on the public internet by mid-2025. They also catalogue device-level deployments from Apple Intelligence to Plaud NotePin. The picture that emerges is one of rapid, chaotic growth: each paradigm has genuine strengths, but each also recreates the same problems of isolation, security weakness, and duplicated effort.

App stores, for instance, let anyone package an LLM into a usable product, but the result is a closed ecosystem. An app built for OpenAI's GPT Store cannot run on Coze or Yuanqi because there are no shared specifications for packaging, skill integration, or data handling. Developers rebuild the same logic from scratch on every platform. Agent frameworks face an analogous problem at the protocol level: LangChain, AutoGen, and CrewAI define their own APIs, memory handling, and tool-calling syntax, so an agent built for one framework cannot talk to agents on another. The authors liken this to the internet before TCP/IP: functional, but fragmented.

Self-hosted services offer data sovereignty, yet a 2025 internet-wide scan found that over 40% of public-facing LLM endpoints used plain HTTP with no authentication, exposing model weights and user queries to anyone who cared to look. On the device side, even optimised models struggle with memory, battery, and latency on consumer hardware. Octopus v2, a 2-billion-parameter on-device model, manages to beat GPT-4 on function calling while cutting context length by 95%, but that kind of engineering remains rare.

The Three-Layer Fix

Rather than propose incremental patches, Hou, Zhao, and Wang argue for a redesign organised into three layers. The infrastructure layer consolidates foundational models (multimodal LLMs, embedding models, code LLMs, diffusion models), compute resources spanning cloud GPUs to edge NPUs, communication networks, and data sources including vector databases. The protocol layer is the linchpin: it defines how agents, tools, and devices discover each other, exchange context, delegate tasks, and verify identity. The application layer is where users interact with planning, perception, memory, and action modules, potentially across devices, vendors, and form factors.

The paper catalogues nine representative agent protocols already in the wild. Anthropic's Model Context Protocol (MCP) standardises tool invocation and context sharing over JSON-RPC. Google's Agent-to-Agent Protocol (A2A) handles peer-to-peer task delegation through Agent Cards. The Agent Network Protocol (ANP) uses W3C decentralised identifiers for open-web agent discovery. IBM's Agent Communication Protocol (ACP) supports asynchronous, MIME-typed messaging for enterprise environments. These are promising starts, but adoption remains fragmented and none covers both tool integration and agent-to-agent communication in a unified way.

Why Security Can't Wait

A recurring theme across all four paradigms is that security is treated as an afterthought. In app stores, automatic screening routinely misses prompt injection and hidden data collection. In agent systems, memory extraction attacks (MEXTRA) let adversaries pull private user data from stored context. Multi-agent setups compound the risk: a compromised agent can propagate malicious instructions to collaborators, and researchers have demonstrated covert collusion via steganographic channels, entirely invisible to oversight tools.

The authors advocate for a secure-by-design approach woven into every layer. At the application layer, that means structured prompt templates (not free text), encrypted memory modules with expiration rules, and shared reasoning-action logs for auditability. At the protocol layer, it means session tokens, typed permissions, consent markers, and reputation systems that can quarantine badly behaving agents. At the infrastructure layer, it means confidential computing, signed model weights, and sandboxed execution for plugins. The default posture should be that every agent is untrusted until verified, and important decisions require cross-checks from multiple agents or models.

What Comes Next

The paper identifies five concrete opportunities. First, protocol-driven ecosystems that extend familiar web technologies (HTTP, gRPC, OAuth) into the LLM domain, with standard bindings to robotics middleware like ROS2 and IoT protocols like MQTT. Second, the secure-by-design practices described above. Third, human-centred monitoring, where dashboards visualise each reasoning step and let users rewind, edit, or veto agent actions. Fourth, device ubiquity, with NPUs and tensor accelerators making on-device LLM inference routine on consumer hardware. Fifth, composable applications, moving from monolithic agents to a plugin marketplace of micro-capabilities (code execution, scheduling, search) that snap together via protocol-level interoperability.

This is a vision paper, not an empirical study. The authors acknowledge they cover representative paradigms rather than an exhaustive taxonomy, and they highlight emerging trends rather than provide a systematic literature review. The three-layer architecture is a conceptual proposal, not a tested implementation. Its value lies in framing the problem clearly: the LLM application ecosystem is growing faster than the standards needed to hold it together. Without shared protocols, open packaging formats, and security-first design, the likely outcome is a patchwork of walled gardens, each reinventing the same wheels while exposing the same vulnerabilities. The internet got TCP/IP. LLM applications need something equivalent, and the clock is ticking.
