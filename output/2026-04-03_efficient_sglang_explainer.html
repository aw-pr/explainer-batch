Research Explainer · Zheng, Sheng et al. (2024)

LLM inference is full of wasted computation; SGLang recycles it for up to 6.4× higher throughput

A co-designed frontend language and runtime that automatically reuses KV cache across chained LLM calls, compresses finite state machines for faster structured output, and treats the whole program (not just a single request) as the unit of optimisation.

The Problem

Modern LLM applications rarely make a single generation call and call it a day. Agents loop through tool calls, chain-of-thought prompts fork into parallel branches, few-shot evaluations repeat the same preamble hundreds of times, and JSON-mode outputs must conform to a schema token by token. The authors call these patterns Language Model Programs: multi-call structures with control flow, structured I/O, and heavy prompt overlap.

Existing inference engines treat each call as an isolated request. They throw away the KV cache the moment a generation finishes, recomputing thousands of shared prefix tokens from scratch when the next call in the same program arrives. Constrained decoding (forcing output to match a regular expression) proceeds one token at a time, even when long stretches of the output are fully determined by the schema. These inefficiencies compound: a four-turn chat recomputes the entire conversation history at every turn, and a JSON field like {"summary": " burns through 11 decoding steps for what could be resolved in one.

How SGLang Works

SGLang is a Python-embedded domain-specific language paired with a custom runtime. The frontend gives you primitives (gen, select, fork, join) that compose naturally with Python's control flow. A decorated function manages prompt state as an asynchronous stream, so forked branches run in parallel without manual threading. The runtime, called SRT, receives these structured calls and applies three key optimisations.

RadixAttention stores KV cache tensors in a radix tree (a compressed trie) with an LRU eviction policy. When a new request arrives, the runtime performs prefix matching against the tree. If the first 2,000 tokens of your 2,200-token prompt already exist in the tree, only the final 200 need a fresh forward pass. The tree splits and merges nodes dynamically as chat sessions branch, few-shot batches arrive, and self-consistency samples fork from a shared prompt. A cache-aware scheduler reorders waiting requests by matched prefix length (longest first), which the authors prove is equivalent to a depth-first traversal of the request radix tree and achieves optimal cache reuse in the offline case.

Compressed finite state machines tackle structured output speed. Standard constrained decoding converts a regex into an FSM and masks invalid tokens at each step. SGLang compresses adjacent single-transition edges into one edge: the constant string {"summary": " collapses from 11 FSM states to a single compressed transition that can be decoded in one forward pass. This yielded a 1.6× throughput increase on JSON decoding alone.

What They Measured

The evaluation covers 11 workloads (MMLU, HellaSwag, ReAct agents, generative agents, tree-of-thought, skeleton-of-thought, LLM judges, JSON decoding, short and long multi-turn chat, and a DSPy RAG pipeline) across Llama-7B, Llama-70B, Mixtral-8x7B, LLaVA image, and LLaVA video models on NVIDIA A10G and A100 GPUs. Baselines include vLLM v0.2.5, Guidance v0.1.8 with llama.cpp, and LMQL v0.7.3 with HuggingFace Transformers.

SGLang achieves up to 6.4× higher throughput and up to 3.7× lower latency on the Llama-7B suite. The gains persist on larger models: Mixtral-8x7B on 8 A10G GPUs and Llama-70B on 4 A100s show similar relative speedups. On multi-modal models, SGLang processes LLaVA-v1.5-7B images at 1.15 images/s versus 0.18 images/s for the authors' original HuggingFace implementation, a 6× improvement. In production deployment on Chatbot Arena, RadixAttention achieved a 74.1% cache hit rate for Vicuna-33B and reduced first-token latency by 1.7× on average.

The ablation study is thorough. Removing the tree structure (using a flat table cache instead) hurts performance. Switching to first-come-first-served or random scheduling degrades cache hits. Disabling frontend parallelism or the frontend "hint" mechanism (which primes the tree before fork branches arrive) also causes regressions. Each component contributes, and the overhead of maintaining the radix tree data structure amounts to less than 0.3% of total runtime on a no-reuse benchmark.

Why It Matters

The conceptual shift here is treating the program, not the individual request, as the unit of inference optimisation. Every serious LLM deployment involves multi-call patterns: agents, RAG pipelines, evaluation harnesses, structured extraction. SGLang's insight is that these patterns create systematic, predictable prefix sharing that a runtime can exploit automatically, without the developer manually wiring up cache hints.

RadixAttention has already been partially adopted by vLLM as an experimental feature, and SGLang itself serves open-weight models in Chatbot Arena. The compressed FSM work sits at the intersection of formal language theory and LLM serving, a space that will grow as JSON-mode, function-calling, and tool-use outputs become standard expectations. The overhead is negligible when there is nothing to reuse, so the system loses nothing by keeping RadixAttention enabled by default.

The limitation the authors flag honestly: cache-aware scheduling can starve low-priority requests, and the compiler mode cannot yet handle data-dependent control flow. The system also does not extend the radix tree across memory hierarchy tiers (GPU memory to DRAM to disk). These are engineering constraints, not conceptual ones. The core argument, that multi-call structure is exploitable and worth exploiting, is well-supported by the numbers.
