Serving LLMs costs far more than the arithmetic alone, and network latency is the reason

A theoretical roofline model for LLM inference reveals that GPU collective latency, not memory bandwidth, is the dominant constraint on cost-per-token. The model produces Pareto frontiers of speed vs. cost that match real API provider pricing within a factor of two.

Published June 2025

~$0.27/M tokens predicted efficient-point cost for Llama 3 70B (8-bit) on H100 at 107 tok/s, matching leading API providers

3-5× gap in hardware utilization between prefill (input tokens) and autoregressive decoding (output tokens) at practical speeds

Why we still lack a roofline model for inference cost

Training a large language model is expensive, but running it afterwards can be worse. Cumulative inference costs across all users now rival or exceed training budgets for popular models. A competitive market of API providers has sprung up to serve open-source LLMs cheaply and quickly, yet there has been no public, quantitative framework for understanding the theoretical limits of what this hardware can deliver.

Erdil's paper fills that gap. It builds a first-principles model that takes a model architecture and a GPU spec sheet as inputs and outputs a Pareto frontier: every achievable combination of serial decoding speed and cost per token, assuming optimal choices of batch size, instance size, and parallelism strategy. The model accounts for arithmetic throughput, HBM bandwidth, network bandwidth, network latency, and the topology of multi-GPU clusters.

The striking result is that the model's predictions match real-world API pricing within a factor of two, without any parameter fitting. Leading providers appear to operate close to the theoretical frontier.

Latency is the real cost driver, not memory bandwidth

The conventional wisdom from hobbyist single-GPU inference is that memory bandwidth is the bottleneck. You can only generate tokens as fast as you can stream weights out of HBM. This is true on one device, but it breaks down completely at scale.

When you spread inference across many GPUs with tensor parallelism, each layer requires all-reduce operations to synchronize intermediate results. Even a small all-reduce on NVIDIA hardware costs roughly 7 microseconds of base latency plus additional time scaling with the square root of the number of participating GPUs. Multiply that by four sequential matrix multiplications per layer and 80 layers, and the latency overhead becomes the dominant term in the cost equation.

This is the key insight of the paper. In theory, you could quadruple serial speed for each doubling of cost by simply throwing 8× more GPUs at the problem, because matrix multiplications are three-dimensional while the tensors being communicated are two-dimensional. But latency grows with instance size, and eventually the all-reduce overhead exceeds the time saved by distributing the computation. There is an optimal instance size for every model, and going beyond it makes inference slower, not faster, at any cost.

What actually moves the cost curve

The paper evaluates three major levers for reducing cost per token: quantization, speculative decoding, and hardware generation. None of them are as powerful as you might hope, and the reasons are instructive.

Quantization helps, but modestly. Halving weight precision from 16-bit to 8-bit reduces cost per million tokens for Llama 3 70B from $0.70 to $0.37, a factor of roughly 1.9×. Going further to 4-bit gets you to $0.23. The simplified model predicts each halving should yield about a 26% latency improvement, and the full model confirms this is approximately right. The big wins come not from the precision reduction itself but from crossing a hardware boundary: fitting a model that previously required two nodes onto a single node, for instance.

Speculative decoding is the more interesting lever. Using Llama 3 8B as a fast approximator for Llama 3 70B at an 80% token acceptance rate yields a 66% throughput increase at constant cost. For Llama 3.1 405B the effect is even larger: a full doubling. The paper notes that acceptance rates between 0.7 and 0.9 are typical for model pairs roughly an order of magnitude apart in size, making this a broadly applicable technique. Providers not using speculative decoding face a structural cost disadvantage.

Across GPU generations, the gains are surprisingly modest. Moving from A100 to H100 improves tokens per second at a fixed cost by about 25%, and roughly halves the required instance size. The paper notes this pattern and leaves open whether B100 will continue the trend.

Long context makes everything more expensive

At short context lengths, the KV cache is negligible and cost is driven by weight reads and network latency. At long contexts, the KV cache dominates everything. Mistral Large 2 at 100K tokens of context requires 36 GB of KV cache reads per generated token, more than the 244 GB model itself at batch sizes above 7. The memory bandwidth cost alone sets a floor of $6.06 per million output tokens on an H100.

Grouped-query attention (GQA) matters enormously here. Without it, the arithmetic intensity of attention during decoding drops below 1% on NVIDIA hardware. Models like DeepSeek-V3, which use multi-head latent attention to compress the KV cache by a factor of roughly 6× relative to Mistral Large 2, pay for this with higher arithmetic cost per attention operation, but the tradeoff is overwhelmingly favorable for long-context inference costs.

The paper also explains why input tokens are typically priced 3× cheaper than output tokens. During prefill, the GPU achieves 50% or better arithmetic utilization because all input tokens can be processed in parallel. During decoding, utilization drops to 10-17%. This roughly 3:1 ratio matches observed pricing at providers like OpenAI.

The inverse square root law and what it means for your budget

One of the cleanest results in the paper is a scaling law for dense models: the maximum achievable tokens per second scales as the inverse square root of parameter count. A model with 8 billion parameters can theoretically hit roughly 960 tokens per second on H100s. Scale that to 70 billion parameters and you get about 234 tokens per second. The ratio is almost exactly the square root of 70/8.

This scaling law is not just an empirical observation. It drops out of the theoretical analysis under reasonable assumptions about how model depth scales with width. It also has a direct cost implication: because the arithmetic cost per token scales linearly with parameters while the achievable speed scales as the inverse square root, the minimum cost per token at maximum speed is roughly proportional to the square root of the parameter count as well.

For anyone budgeting inference costs, the practical message is clear. Going from a 70B to a 405B model does not multiply your per-token costs by 405/70. But it does multiply them by something closer to the square root of that ratio, roughly 2.4×, before accounting for architectural differences like MoE sparsity. The actual predictions from the full model ($0.27 for Llama 3 70B vs. $1.31 for Llama 3.1 405B) track this surprisingly well.

BOTTOM LINE

LLM inference cost is not primarily an arithmetic problem. It is a latency problem. Network and NVLink latency from GPU collective operations set the floor on how cheaply you can serve a model at any given speed. Speculative decoding and quantization can each cut that cost by up to 2 times, but the fundamental constraint is the topology of the GPU cluster and the microseconds lost to each all-reduce. The paper's theoretical model matches real-world API pricing without calibration, suggesting that leading providers are already operating near the theoretical frontier.
